# Git Push PR

## 目标

把当前工作从本地仓库安全地推进到远端评审流程，默认包含提交、推送和创建 Pull Request。

## 工作流

1. 检查分支与工作区状态

- 查看当前分支名、`git status --short`、远端信息和默认目标分支。
- 如果处于 detached HEAD，先停止并说明无法直接推送或建 PR。
- 如果存在未提交改动，先按 `$git-commit` 的流程整理提交，再继续。

2. 确认推送前提

- 确认当前分支不是长期主干分支；如果用户正在 `main`、`master` 或其他受保护主干上开发，优先创建功能分支再继续。
- 推送前再次确认没有把无关改动混进当前分支。
- 若远端或认证不可用，记录阻塞点，并给出下一步所需命令或条件。

3. 推送分支

- 使用非交互命令推送，并在需要时设置 upstream。
- 不要在未明确授权时执行强推或重写远端历史。

4. 创建 Pull Request

- 如果仓库托管在 GitHub 且 `gh` CLI 可用，自动创建 PR。
- PR 标题默认与最新提交主题保持一致，并在必要时稍作润色。
- PR 描述至少包含：改动摘要、验证方式、风险或待确认项。
- 若无法自动创建 PR，输出建议标题与描述，方便用户后续手动创建。

5. 返回结果

- 汇总最终分支、推送结果、PR 链接或失败原因。
- 如果只完成了其中一部分，也要明确说明停在了哪一步。
